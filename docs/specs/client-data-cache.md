# Client data cache — "instant tabs" (Phase 6 · Slice B)

**Status:** v4 — piecewise restructure + builder pre-build sweep folded in (supersedes v3 + the v3.1 honesty patch), 2026-07-24 · **Author:** REVIEWER, with the owner · *v4 edited by BUILDER at owner instruction; every modification/addition/deletion carries an italic reason in place*
**Parent plan:** [future-plans.md §App feel & performance](../future-plans.md) / PLAN.md Phase 6 item ② ("cache everything except the data" — owner 2026-07-11)
**Ground rule:** FE-only. No DB change, no RLS change, no new server privileges. The server stays the only source of truth.

## Problem

Every tab tap re-runs the full server pipeline (auth round-trip + role query + the page's DB queries) before anything paints. Skeletons (Phase 6 ①) made the wait *visible*; nothing yet made it *short*. A revisited tab knows nothing from the last visit.

## Approach — one committed layer, one conditional layer, three pieces

*(v4 — restructured from "two layers, four commits": the pre-build sweep showed that ALL of the experimental risk lives in Layer 1 alone — it hangs on `experimental.staleTimes`, whose own docs (checked 2026-07) still say "not recommended for production" — while Layer 2 is entirely stable, officially-documented machinery. The owner asked for standalone pieces, the first being medium effort/risk/win. So: Layer 2 ships first as two self-contained pieces; Layer 1 is quarantined into a final optional piece with its own go/no-go that the owner can decline indefinitely without weakening pieces 1–2.)*

**Layer 2 — data cache (TanStack Query v5) · Pieces 1–2, committed.** The hot lists move their data into client queries seeded by the server render. On every mount the cached list paints **immediately** and a background refetch fires; fresh results are applied **in place** (rows appear/change/vanish; counts tick — no reload, no flash). This is the classic *stale-while-revalidate* pattern.

**Layer 1 — screen snapshot (Next.js router cache) · Piece 3, conditional.** Configure `staleTimes` so a *client-side* revisit to a recently seen page repaints the **last rendered screen instantly** instead of waiting for the server. In-memory, per-tab, gone on hard reload. Gated on a spike proving the flag's name + semantics on our exact Next version *by experiment*; the window value comes out of the spike feeling right, not out of this spec asserting it. **Plan C if the spike fails** *(named in v3.1, kept)*: restructure the hot list pages to a thin server shell (auth only) + client-fetched data, which makes Layer 2 carry the whole win — a bigger change, specced separately if ever wanted. *(v4 — v3 asserted `dynamic = 300`; the number is now a spike output and the whole layer is opt-in. v3.1's parenthetical "TanStack ignores `initialData` once a cache entry exists, so the per-nav server fetch mostly feeds cold loads" is **superseded** by the new D2: under hydration, every per-nav server render feeds the cache — those fetches are no longer wasted.)*

**Honest UX accounting** *(v4 — promoted from v3.1's caveat into its own section; the piecewise cut only works if each piece's win is labeled truthfully)*: without Layer 1, every in-app tab tap still pays the server round-trip (RSC re-render + its fetch) and shows the skeleton, exactly as today — Pieces 1–2 remove the *data* wait, not the *navigation* wait; a half win on taps. But the highest-frequency pain — flip to another app, flip back — involves **no navigation at all**: the page is still mounted, and what was broken there was a deaf Realtime channel plus no refetch-on-return. Piece 1 fixes precisely that with stable machinery. Back/forward navigation also repaints instantly from the router's back-cache regardless of the flag. What Piece 3 adds — and all it adds — is instant *forward* tab taps.

### The pieces

*(v4 — new section; the governing structure of the build.)*

| Piece | Scope | Effort · risk · win |
|---|---|---|
| **1 — orders that are never stale** | Provider foundation + D9 sign-out hardening + all four orders lists onto the cache + Realtime simplified to invalidate-only | medium · medium · medium — the app's center of gravity becomes permanently self-correcting |

## Decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Library: TanStack Query v5** (`@tanstack/react-query`) | The standard for exactly this job: caching, background refetch, invalidation, focus-refetch — all built in, small, no server component. SWR would also work; TanStack's invalidation ergonomics are better for our write-heavy flows. *(v4 note: `@supabase-labs/tanstack-db` — live-synced collections — surfaced in the builder's research; rejected for v1 as too young for a prod distributor app.)* |
| D2 | **Server render stays; it seeds the cache via `prefetchQuery` → `dehydrate` → `<HydrationBoundary>`** — the official TanStack App Router pattern — *not* `initialData` props. **Security rule: the server-side QueryClient MUST be created per request — never a module-level singleton.** A shared server instance would dehydrate one user's rows into another user's HTML. This is the single place in this slice where a mistake is a security bug, not a staleness bug. | *(v4 — changed from v3's `initialData` + `initialDataUpdatedAt`: the code sweep found 16 `router.refresh()` sites implementing the app's post-write reconcile idiom — optimistic overlay, then refresh delivers fresh server props (ProductsPricing, UsersAdmin, RetailersQueue, DepositsView, PickScreen, DepositFlow). Under `initialData`, a migrated list ignores refreshed props (TanStack reads initialData only at cache-entry creation) — every reconcile flow silently dies and read-migration forcibly couples to write-migration. Hydration merges by `dataUpdatedAt` (newest wins), so `router.refresh()` keeps feeding the cache: reads migrate page-by-page while writes stay untouched until the Piece-2 sweep. The server-fetch timestamp rides the dehydrated payload natively, preserving the no-cold-load-double-fetch property `initialDataUpdatedAt` existed to provide — that mechanism and its clock-skew cap are deleted as no longer needed. The per-request security rule is the REVIEWER's addition.)* |
| D3 | **→ Piece 3, conditional.** Router `staleTimes.dynamic` (value from the spike) — paired with D11, because the knob is global and cannot exclude routes. Without it, App Router re-blocks on the server for every revisit; with it, D5 pages would show stale money screens unless D11 guards them. | *(v4 — moved out of committed scope: the flag is experimental, its docs say not production-recommended, and the owner declined the gamble. Everything v3 said here remains true; it just lives behind Piece 3's gate now, and the asserted `= 300` value is deleted in favor of a spike-chosen one.)* |
| D4 | **Cached lists (per-key):** `["orders", scope]` (scope ∈ `salesman` · `staff` · `godown-home` · `godown-dispatch`), `["products", scope]`, `["retailers"]`, `["deposits", scope]`, `["catalog"]`. The catalog is safe to cache: `submit_order` re-prices fixed brands server-side, so a stale picker price cannot produce a wrong order. **Standing assumption (write it in code comments too):** these coarse keys are only correct because every list is a *bounded fetch-all with client-side filtering* (orders `.limit(300)`, deposits `.limit(500/1000)`, products/retailers/catalog uncapped-under-row-cap — builder-verified). If any list ever moves to server-side filtering/pagination, its key MUST grow the filter params. | *(v4 — three amendments from the code sweep. **(a)** The salesman orders query bakes the user id into its SQL — `.or('status.neq.cancelled,cancelled_by.neq.<uid>')` — so builders take uid as a parameter, and uid-less keys are safe ONLY because D9 wipes the cache on every auth transition; that dependency was implicit and load-bearing, now explicit, and it gets a comment at the builder. **(b)** `["retailers"]` stays ONE key only via a shared **superset builder** — the dashboard queue's columns (`id,name,area,phone,verified,active`) cover the Quick-Order picker's (`id,name,area,verified`); two different selects under one key would drift, and the extra bytes are harmless at ~600 rows. **(c)** The **godown pickup queue** (`/godown`) was in neither D4 nor D5 — a blind spot, and it's the godown's hottest screen. It stays a pure always-fresh server page in Pieces 1–2 (the safest possible state; `submit_pick` re-validates server-side so a stale queue cannot corrupt a pick) and gets its snapshot decision alongside Piece 3.)* |
| D5 | **Never cached:** order **detail**, deposit void/edit state, the PDF route, anything inside the approval/billing flow, auth/role. Money decisions read fresh, full stop. (PDF already sends `Cache-Control: no-store`.) **Concrete guard set for Piece 3** — the pages that get D11 when, and only when, Layer 1 turns on: `/orders/[id]`, `/dashboard/orders/[id]`, `/godown/orders/[id]`, `/godown/[id]`, `/scan/[id]`, **`/new-order?edit=<id>`**, `/deposits/new`, `/dashboard/users`. | *(v4 — v3 said "every D5 page" without a list; hand-waving is how holes survive, so the set is now enumerated from the actual route tree. The sharpest addition is `?edit=`: the editor seeds its form from `order_items` server-side, so a snapshot-old payload could seed superseded lines and a save would silently revert a concurrent edit — stale display becoming stale writes. The plain `/new-order` and the `?edit=` variant are distinct router-cache URLs (spike must confirm), so they can be treated differently. `/deposits/new` is included because its recent-deposits fetch feeds duplicate detection — a money guard. **In Pieces 1–2 none of this apparatus exists or is needed:** no Layer 1 → no stale snapshots → the hazard class arrives with the flag, bundled with its countermeasures. The related pre-existing lost-update gap is parked separately — see Out of scope.)* |
| D6 | **`staleTime` ≈ 5–10s on every cached list** (a rapid-flip dedupe, not a freshness budget) + `refetchOnMount: true` (default, stale-triggered) + focus and reconnect refetches **via TanStack's built-in `focusManager` / `onlineManager`** (`refetchOnWindowFocus`, `refetchOnReconnect`). | Owner call: *every* visit re-asks the server — the page just never waits for the answer. With D2's hydration timestamp, the first mount reads as fresh (no duplicate); any revisit older than ~5–10s refetches immediately in the background. *(v4 — added the managers clause: the custom `visibilitychange`/`online` handlers from `05c3bd0` are **deleted** in Piece 1. They were the right stopgap pre-cache; post-cache they are re-implementations of triggers the library ships, and two implementations of the same trigger double-fetch and drift.)* |
| D7 | **Writes invalidate:** every mutation (submit/edit/cancel order, approve, punch, product save, deposit create/void, retailer edit) calls `invalidateQueries` for its affected keys. Existing `router.refresh()` calls stay **and remain functional**. | The actor sees their own change immediately; other devices converge via dedupe windows + Realtime + focus-refetch. *(v4 — corrected claim: v3 called the surviving refresh calls "harmless overlap," which would have been false under `initialData` (dead letters) and is true **by construction** under hydration — a refresh re-renders the server page and its dehydrated payload feeds the cache. Pruning stays deferred (⑦). Scheduling note: Piece 1 carries the order-RPC invalidations only; the full cross-surface sweep is Piece 2's second commit — a decoupling made safe by exactly this D2 property.)* |
| D8 | **Memory-only cache — no disk persistence** in v1 | Closing the app = clean slate, zero cold-start staleness class. (An offline-tolerant `persistQueryClient` layer is a possible v2, deliberately out of scope; the cold-open instinct is served properly by Slice C.) |
| D9 | **Any transition to signed-out wipes everything** — `onAuthStateChange(SIGNED_OUT)` → `queryClient.clear()` + hard navigation (`await signOut()` **then** `location.assign("/login")` — that order), **plus a `pageshow` handler that reloads when `event.persisted`** | Owner call: auto sign-outs must wipe like the button. Builder-verified: today's SignOutButton is a *soft* `router.push` — the hard-nav is genuinely missing, add it. The `pageshow` buster covers iOS/WebKit back-forward cache, which can restore the previous page from memory with zero network after a hard nav (`no-store` does not reliably prevent it on iOS — and this is a phone PWA). *(v4 — content unchanged, scheduling changed: promoted into Piece 1's foundation commit as standalone security hardening worth shipping regardless of any caching.)* |
| D10 | **Realtime handlers do exactly ONE thing: `invalidateQueries(["orders", …])`.** No per-row `setQueryData`, no hand-specified convergence rule. The `newIds` "new row" highlight stays driven by the event payload (presentation state, not data). The migrated OrdersView renders **only** from `useQuery`; the `useState(initialOrders)` + render-phase re-seed from `05c3bd0` are removed in the same commit *(superseded: re-seeding by hydration, the visibility/online net by D6's managers)*. Pre-slice fixes stand (`eba5311` setAuth-before-subscribe + refresh-token follow; status-aware subscribe with retry; on recovery, invalidate instead of `router.refresh()`). Nothing may *depend* on the socket. | *(v4 — simplified from v3's setQueryData-plus-convergence-rule, on two grounds. **Scale:** at <20 orders/day against a bounded 300-row list, one refetch per event is negligible — per-row surgical patching is complexity with no payoff. **Library:** `invalidateQueries`' default `cancelRefetch: true` already cancels a stale in-flight response and reissues a fresh one — v3's convergence rule hand-specified behavior the library performs by default, so the rule text is deleted. One code path that always converges beats two refereed by a rule; this is also the community-standard Supabase+TanStack integration (Realtime as an invalidation signal, not a data writer). Deflates "the hard commit" considerably.)* |
| D11 | **→ Piece 3, conditional.** `<RefreshIfStale renderedAt={…} />` on every page in D5's guard set — a tiny client component: on mount, if `Date.now() − renderedAt > ~10s`, call `router.refresh()` once. A snapshot money screen paints instantly, then re-renders fresh in ~1s; the ~10s age guard means genuinely fresh first visits don't double-render and absorbs phone clock skew. | *(v4 — moved with D3: without Layer 1 there are no stale snapshots to refresh, so the component ships only alongside the flag. Mechanics unchanged from v3.)* |
| D12 | **Shared query builders — each list query exists ONCE** in `src/lib/queries/`, parameterized by the Supabase client (and uid where the SQL needs it — D4a), imported by BOTH the server page and the browser `queryFn`. Same embeds, same sort, same caps, by construction. **Builders land in the same commit as their first consumer.** | Duplicated query logic drifts, and drift here means rows flash/vanish/reorder ~1s after every visit, forever. Per-list verification task (not an assumption): every role's list query must be proven to return identical rows under the *browser* client's RLS as under the server client — acceptance P1-5 / P2-2. *(v4 — added the first-consumer rule: v3's plan put the builders in the foundation commit, where they'd be dead code; this repo's review loop verifies by execution, and dead code reviews poorly.)* |
| D13 | **A failed background refetch never blanks a painted list** — components render from data presence (`data` exists → render it), never gate on `isError`; errors surface as a quiet retry affordance at most. | Salesman networks are flaky; a refetch failure after a successful paint must degrade to "slightly stale," not to an empty screen. *(v4 note: hydration makes this trivially satisfiable — a mounted list always has seed data — but the guard still gets written, because a future code motion could break the invariant silently.)* |

## What the user experiences

*(v4 — expanded from the single 500→502 case: each piece's headline flow is stated, honestly labeled.)*

**The flip-away-and-back ritual (Piece 1):** backgrounding the PWA and returning triggers a focus refetch; the still-mounted orders list corrects in place within ~1s, socket dead or alive — the daily-pain case, fixed without any experimental flag.

**The 500 → 502 case (Piece 2):** reopen Products: the cached 500 rows paint instantly → background refetch lands (~0.5s) → header ticks to 502 and the two new rows slot into their sorted positions. Unchanged rows are not re-rendered (React keys by id) — no flicker.

**Tab taps (until Piece 3):** an in-app tab tap still pays the server round-trip and shows the skeleton, exactly as today. The "instant taps" promise belongs to Piece 3 alone.

## Interplay with what already exists

- **loading.tsx skeletons** remain — under Pieces 1–2 they appear exactly as today; under Piece 3 they'd cover only first visits/hard loads.
- **sessionStorage filter persistence** (orders) is orthogonal and untouched — filters restore, data comes from cache.
- **Security invariants:** RLS still scopes every fetch (browser client uses the user's session); RPCs still re-check roles server-side; prices/totals still recomputed server-side at submit. A cache never bypasses any of it — it only remembers what this user was already allowed to see.
- **sw.js is a pure network passthrough** (builder-verified) — no interference with either layer. *(v4 added — checked during the sweep; recorded so nobody re-checks.)*
- **`createBrowserClient` (@supabase/ssr) is singleton-by-default** — one global `onAuthStateChange` listener for D9 is safe plumbing. *(v4 added — verified during the sweep.)*

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Stale flash (seconds-old list shown briefly) | By design; bounded by D6 + in-place correction; money surfaces excluded (D5). |
| Cached data outliving a session on a shared phone | D9 wipe-on-any-signout; caches are per-tab memory only (D8); `pageshow` buster for WebKit bfcache. |
| **Server-side QueryClient leaking rows across users** | D2's bold rule: per-request creation, never a module singleton. *(v4 added — the one security-class risk in the slice.)* |
| **Scroll shift when rows insert above viewport** | *(v4 — corrected from v3's "rare, mild — revisit if reported": on the orders list new rows insert at the **top**, which is above the viewport whenever the user has scrolled at all — per-event on the busiest list, not rare. Chrome's built-in scroll anchoring compensates; iOS Safari historically doesn't. Acceptance P2-4 requires the real-iPhone test before sign-off; if it fails there, mitigations (anchor pinning, a "N new orders" pill) become their own small decision.)* |
| Library bloat | ~13kb gzipped, no server piece; devtools dev-only and tree-shaken from prod. |
| Phone clock skew vs freshness math | `dataUpdatedAt` rides the dehydrated payload; D11's threshold (Piece 3) stays generous (~10s). *(v4 — the `Math.min` client cap is gone with `initialDataUpdatedAt`; skew in either direction now costs at most one harmless extra background refetch.)* |
| **Piece 3 only — `staleTimes` is experimental** | Docs still say "not recommended for production" (checked 2026-07). The spike is the gate; declining Layer 1 costs Pieces 1–2 nothing; Plan C is the named alternative. *(v4 added.)* |
| **Piece 3 only — the snapshot window bypasses the auth proxy** | Within the window, tab flips never hit `src/proxy.ts`, so an expired session lingers until a Layer-2 refetch 401s and D9 fires — the two layers back-stop each other. *(v4 added — found in the sweep; recorded verbatim per REVIEWER.)* |

## Out of scope (separate slices)

Slice A (role in session claims), Slice C (service-worker shell caching + update strategy), Slice D (⑭ FK indexes), disk persistence, offline mode, `router.refresh()` pruning (⑦), Plan C (thin server shell — only if Piece 3's spike fails AND instant taps are still wanted).

**Parking lot — edit-conflict guard** *(v4 added per REVIEWER)*: the order editor has no conflict check today — anyone sitting on an open editor for minutes can already silently revert a concurrent edit; Layer 1 would have *widened* a window that already exists. A "someone else changed this order while you were editing" guard (e.g. an `updated_at` captured at form-seed time and compared inside `update_order_items`) is its own future slice with its own spec.

## Acceptance (REVIEWER verifies by execution)

*(v4 — re-bucketed per piece; v3's single 12-point list assumed everything shipped at once. Every v3 item survives, re-homed; the only new items are P1-8 and P2-4.)*

**Piece 1 — orders + foundation:**
1. Flip the PWA away and back → the orders list corrects in place ≤ ~1s (focus refetch), socket dead or alive.
2. Add an order on device A → device B's list updates within ~1s (Realtime → invalidate), with the new-row highlight.
3. A Realtime event during an in-flight refetch converges to server truth — no row flickers back to a dead state.
4. Cold load fires exactly ONE orders fetch (server prefetch only — no client duplicate; network tab).
5. Per role (salesman / staff / godown), the browser-client refetch of each orders scope deep-equals the server payload — same rows, same order.
6. Sign-out (button AND expired-session path) wipes everything; back button cannot show any cached screen — **verified on Chrome AND iOS Safari** (the `pageshow` buster is what makes Safari pass).
7. Kill the network after a successful paint → the list stays painted; no blank screen.
8. Every order RPC (approve / cancel / process / dispatch / punch / step-back / edit / submit) is reflected in the actor's own lists without a manual reload. *(v4 added — Piece 1 carries the order-key invalidations.)*
9. `tsc` / `eslint` / `build` clean; zero DB/RLS diff; first loads byte-match today's render.

**Piece 2 — remaining lists + sweep:**
1. The 500→502 Products scenario behaves as described (count ticks, rows slot in, no reload).
2. P1-4 / P1-5 / P1-7 repeated for products (both), catalog, retailers (superset builder on both surfaces), deposits (both scopes).
3. Every mutation in the D7 table refreshes every affected surface for the actor without a manual reload — and the existing `router.refresh()` reconcile flows (ProductsPricing stock/price, RetailersQueue, DepositsView void, PickScreen, DepositFlow) still work unchanged. *(v4 — the "unchanged" clause is the D2 decoupling made testable.)*
